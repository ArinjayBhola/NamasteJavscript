If we have thousands of lines of code that take 10 seconds to execute, and within that code, there is a setTimeout with a 2-second timer, here's what happens:

The global execution context (call stack) will be busy executing the code for the full 10 seconds. Although the setTimeout has a 2-second timer, the callback function associated with it will not be executed immediately after 2 seconds. Instead, once the 2-second timer expires, the callback function is placed in the callback queue.

However, because the call stack is still processing the synchronous code, the event loop, which constantly monitors the call stack, will wait until the stack is empty before pushing the setTimeout callback onto the call stack. This means the setTimeout callback will only be executed after the entire 10 seconds of synchronous code execution is complete. The event loop ensures that the callback function is eventually executed, but only after all other synchronous code has finished running.
